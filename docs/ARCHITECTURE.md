# 架构说明

这一页只回答两个问题：请求从哪里进来、费用和状态最后落到哪里。具体的环境变量和上线步骤请看 [部署手册](DEPLOYMENT.md)。

## 组成

```text
客户端 / SDK / CLI
        │  Anthropic、OpenAI、Gemini、xAI 兼容 API
        ▼
Gin 网关（鉴权、模型映射、协议适配、用户 / 密钥并发槽）
        │
调度与账号池（账号并发槽、分组、优先级、冷却、代理）
        │
上游平台 API

控制台（Vue 3） ────────► 管理与用户 API ────────► SQLite / PostgreSQL
```

## 请求链路

1. 网关从请求头读取 `dd-` API Key，并校验密钥状态、额度、已选分组和调用限制。旧的 `group_id` 保留为默认分组，多分组关系由 `api_key_groups` 维护并在升级时自动回填。
2. 根据兼容端点和模型目录确定目标平台及上游模型名。
3. 请求原子占用用户和密钥并发槽，再由调度器占用一个可用账号槽；槽位已满时进入有界等待队列。
4. 网关先按模型目录与请求协议选择匹配平台，再由调度器从对应分组选择可用账号；账号池为空、排队拥塞或发生可重试上游错误时，先尝试下一账号，再切换到同平台的下一个已选分组。
5. 请求经可选出站代理转发给上游。流式响应以 SSE 形式透传，同时提取最后的 usage 数据。
6. 服务按模型价格、输入/输出/缓存 Token、用户倍率和分组倍率结算，并记录用量、账单与运行指标。

## 并发控制

- 用户并发是该用户所有密钥共享的总上限；密钥并发可以再收紧单个凭据；账号并发保护单个上游凭据。
- `0` 表示该层不限制，便于旧数据平滑升级。三个槽位都会覆盖请求的完整生命周期，包括流式响应。
- 等待时间和全局队列深度由“系统设置 → 网关策略 → 并发保护”控制。队列满或等待超时返回 `429` 与 `Retry-After`，不会继续堆积请求。
- 客户端断开、上下文取消、上游失败、重试切换和流结束都会释放对应槽位。排队耗时写入用量日志，并显示在运行监控中。
- 当前槽位存储与账号调度器一致，都是单进程内存状态，适合现有的单网关部署。扩展为多网关副本前，需要将槽位计数与唤醒机制替换为 Redis 等共享实现，不能依赖各实例独立计数。

## 代码边界

| 目录 | 职责 |
| --- | --- |
| `backend/cmd/server` | 服务启动入口 |
| `backend/internal/config` | YAML 与环境变量配置 |
| `backend/internal/model` | GORM 数据模型 |
| `backend/internal/store` | 数据库迁移与初始化 |
| `backend/internal/gateway` | API 鉴权、协议适配、转发、流式与 usage 提取 |
| `backend/internal/service` | 调度、计价、支付、邮件、告警与运行指标 |
| `backend/internal/handler` | 控制台、用户端和管理端 HTTP API |
| `frontend/src` | Vue 控制台、主题、组件与 API 客户端 |
| `deploy` | Docker、systemd 与 Nginx 部署模板 |

## 数据与安全边界

- 平台 API Key 只保存不可逆摘要；明文只在创建时返回一次。
- 上游账户凭据使用 AES-GCM 加密后存储，密钥来自 `ENCRYPTION_KEY`。
- 支付配置、OAuth 凭据、数据库及备份均属于敏感数据，不应出现在浏览器端、日志或版本库。
- 前端只负责控制台展示；鉴权、结算和账户选择都在后端完成。
