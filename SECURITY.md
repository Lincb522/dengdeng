# 安全说明

这份清单用于发布前自查。只要涉及密钥、数据库、支付或 OAuth，就按敏感数据处理。

## 凭据与数据

- 真实的 API Key、OAuth token、支付商户密钥、数据库、证书和 `.env` 文件不得提交到仓库。
- 生产环境必须设置独立且随机的 `JWT_SECRET` 与 `ENCRYPTION_KEY`。可用 `openssl rand -hex 32` 生成。
- `ENCRYPTION_KEY` 一旦用于加密上游凭据，不应直接更换；轮换前必须先完成凭据迁移，否则历史加密数据无法解密。
- 仅使用 HTTPS 对外暴露服务；反向代理应限制管理入口的访问范围，并保留足够的请求体上限以支持 Responses 和图像请求。
- 备份 SQLite 或 PostgreSQL 数据时，按敏感数据处理，并验证恢复流程。

## 漏洞反馈

请不要在公开 Issue、日志或聊天记录中附上可用密钥、完整请求体或用户数据。若发现可被利用的问题，请联系项目维护者并说明：

1. 受影响版本与部署方式；
2. 可复现步骤和最小化示例；
3. 影响范围与建议修复方式；
4. 是否已包含任何凭据（如有，请立即轮换）。

维护者会先确认、修复和部署，再决定是否公开问题细节。

## 上线前检查

- [ ] `.env`、配置文件、证书和数据库未被 Git 跟踪
- [ ] 生产管理员密码已更换，注册策略符合业务需求
- [ ] HTTPS、回调地址和 `SITE_PUBLIC_URL` 均为最终域名
- [ ] 支付、OAuth 与邮件服务使用独立的生产凭据
- [ ] 已验证备份、健康检查、日志轮转与恢复流程
- [ ] 已确认上游账号和服务使用方式符合相应服务条款
